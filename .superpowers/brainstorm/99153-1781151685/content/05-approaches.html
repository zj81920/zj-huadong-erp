三种简化方案对比

基于你的需求：AI 全流程 + 日期简化 + 保留手动进度%

A

渐进增强：保留框架，AI 外挂

数据模型不动，只在现有 4 级 WBS 上加 AI 能力层：AI 生成 WBS 模板、AI 状态判断简化、AI 报表。actualStart/End 字段保留但不强制填。改动最小。

优点

  • 改动小，风险低
  • 现有功能不受影响

缺点

  • 计算逻辑仍有两套
  • 旧字段成为技术债
B

AI 原生：重建数据模型和逻辑

彻底简化：去掉 actualStartDate/actualEndDate/delayDays/status(枚举)，只保留 planStartDate/planEndDate/progress。所有状态和延误由前端纯函数实时计算，AI 只负责 WBS 生成和报表。

优点

  • 模型极简，理解容易
  • 无冗余字段

缺点

  • 需要数据迁移
  • 改动范围大
C

（推荐）务实简化：精简模型 + AI 能力

数据模型精简掉冗余字段，但不废弃现有 API。新逻辑用前端纯函数计算状态（不再有 judgeTaskStatus 和后端 cascadeSummarize 两套）。AI 聚焦在 WBS 自动生成 + 智能预警 + 报表。

优点

  • 模型干净，逻辑单一
  • 改动可控，测试覆盖

缺点

  • 需要数据迁移脚本
  • API 签名需适配
