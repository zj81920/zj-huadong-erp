设计 Section 2：简化状态计算

告别 judgeTaskStatus 五态判断，用纯函数一目了然

❌ 现状：复杂判断链
judgeTaskStatus(task)
  ├─ 已完成(100%) → 还要判断是否超期
  ├─ 未按时开始 → 对比 actualStart vs planStart
  ├─ 等待开始 → planStart 在未来
  ├─ 进行中 → planPct vs progress (5%容忍度)
  └─ delayDays → 反推公式

cascadeSummarize()
  └─ 后端递归汇总写入 status 字段

结果：前后端两套逻辑，5种状态名
✅ 新方案：纯函数计算
只存 progress%，其他实时算

L4 任务状态：
  未开始: progress = 0
  进行中: 0 < progress < 100
  已完成: progress = 100

延误判断（仅进行中任务）：
  planPct = today在计划区间的位置%
  if progress < planPct → 延误

上级节点汇总：
  从子节点(L4)聚合
  → 状态 + 是否延误

核心公式（一个函数搞定所有）

planPct = (today - planStart) / (planEnd - planStart) × 100
状态 = progress=0 ? 未开始 : progress=100 ? 已完成 : 进行中
延误 = 状态=进行中 AND progress < planPct

三个典型场景

✅
按期

计划 1月~10月(300天)
今天 6月(第150天)
planPct = 50%
实际进度 = 55%
55% ≥ 50% → 正常

⚠️
延误

计划 1月~10月(300天)
今天 6月(第150天)
planPct = 50%
实际进度 = 30%
30% < 50% → 延误

⏳
等待开始

计划 8月~12月
今天 6月
progress = 0
未到计划开始日期
