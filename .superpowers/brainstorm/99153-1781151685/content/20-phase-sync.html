阶段变更 & 删除项目 → WBS 响应机制

现有逻辑 + 发现的问题 + 修复方案

三种场景的现有行为

➕ 增加阶段

新阶段名不在已有 L1 节点中 → 自动创建 L1 节点
✅ 正常运作

➖ 减少阶段

L1 节点名不在新阶段列表中 → 直接 delete
⚠️ 若该 L1 下有子节点，delete 会因 FK 约束失败！

🗑 删除项目

deleteMany 删除所有 L1 节点
⚠️ 同样问题：父节点删不掉因为有子节点引用

问题根因

Prisma schema 中，parentId 的自引用关系没有设置 onDelete: Cascade。当删除 L1 节点时，如果它下面有 L2→L3→L4 子树，外键约束会阻止删除。

项目中 已经有一个 deleteChildren() 递归删除函数（在 WBS 节点 DELETE API 里），但阶段同步和项目删除这两个地方没用它。

修复方案（新设计中一并处理）

方案：统一递归删除工具

提取公共的 deleteWbsSubtree(nodeId) 函数，深度优先删除所有子孙节点。减少阶段、删除项目时统一调用。

额外防御：Schema 加 Cascade

在 Prisma schema 的 parent 关系上加 onDelete: Cascade，确保即使 API 层遗漏，数据库层面也能级联删除。

修复后的行为总览

场景WBS 响应风险
➕ 增加阶段创建空 L1 节点，用户后续手动添加 L2-L4无
➖ 减少阶段级联删除该 L1 及其下全部 L2-L4 子树数据不可恢复
🗑 删除项目级联删除所有 WBS 节点 + 任务日志数据不可恢复
✏️ 重命名阶段更新 L1 节点 name无
