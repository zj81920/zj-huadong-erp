# 审批流程系统设计说明

## 一、系统概述

审批流程系统支持管理员在"流程设置"界面自由配置各业务模块的审批节点和角色，无需硬编码。系统根据配置自动驱动审批流转、会签、归档和支付。

### 核心设计原则

| 原则 | 说明 |
|---|---|
| **配置驱动** | 流程节点和角色完全由管理员在界面配置，代码不预设默认流程 |
| **第一节点 = 发起节点** | 无论第一节点配置什么角色，都由提交申请的人自动完成 |
| **发起权限控制** | 第一个节点的角色决定谁有权限发起该流程 |
| **会签机制** | 同一角色下的多个用户必须全部审批通过（会签）；多角色间任一角色完成即推进（或签） |
| **管理员兜底** | admin 角色用户可强制推进卡住的流程节点 |

## 二、流程配置

### 2.1 配置入口

**路径**：系统设置 → 流程设置（`/settings/approval-flow`）

### 2.2 业务模块

系统支持以下业务模块的审批流程配置：

| 模块类型 | 业务名称 | 是否分项目级/公司级 |
|---|---|---|
| `quotation` | 商务报价 | 否（common） |
| `outsourcing` | 外包任务 | 否（common） |
| `purchase_request` | 采购需求 | 否（common） |
| `income_contract` | 收入合同 | 是（project/company） |
| `expense_contract` | 支出合同 | 是（project/company） |
| `non_contract_income` | 非合同收入 | 是（project/company） |
| `non_contract_expense` | 非合同支出 | 是（project/company） |
| `payment_application` | 付款申请 | 否（common） |
| `expense_report` | 费用报销 | 是（project/company） |
| `other_borrowing` | 其他借款 | 否（common） |
| `lending_out` | 借出款 | 否（common） |
| `salary_payment` | 工资发放 | 否（common） |
| `borrowing_return_application` | 借入资金归还 | 否（common） |
| `delivery_receipt` | 送货单 | 否（common） |

### 2.3 节点配置

每个节点包含以下属性：

| 属性 | 说明 |
|---|---|
| **节点名称** | 自定义，如"总经理审批"、"财务审核" |
| **角色** | 支持多选，存储为逗号分隔的角色码（如 `finance,cashier`） |
| **节点序号** | 从 1 开始递增 |
| **节点类型** | `approval`（审批）/ `archive`（归档）/ `payment`（支付） |

### 2.4 操作说明

- **添加节点**：点击"添加审批节点"按钮，新增一个空白节点
- **编辑节点**：直接在节点卡片上修改名称和选择角色
- **删除节点**：点击节点右上角的 × 按钮
- **保存**：点击"保存流程"按钮，保存到数据库
- **清空流程**：点击"清空流程"按钮，清除当前配置的所有节点

## 三、审批流转规则

### 3.1 节点角色规则

| 场景 | 规则 | 说明 |
|---|---|---|
| **第一节点** | 自动跳过 | 无论配置什么角色，都由提交申请的人自动完成（记录 `initiate` 动作） |
| **第一节点角色** | 发起权限控制 | 只有匹配第一节点角色的用户才能发起该流程 |
| **多角色** | 或签 | 多个角色中任一角色的所有用户完成即推进 |
| **同角色多用户** | 会签 | 该角色下的所有用户都必须审批通过 |

### 3.2 会签 + 或签组合示例

假设某节点配置了 `部门负责人,项目经理` 两个角色：

- 部门负责人：张三、李四
- 项目经理：王五

**流转过程**：
1. 张三审批通过 → 部门负责人角色未完成（李四还没批）
2. 李四审批通过 → **部门负责人角色完成** → 立即推进到下一节点（或签，不需要等王五）

如果部门负责人只有张三一人：
1. 张三审批通过 → 部门负责人角色完成 → 立即推进

### 3.3 终节点类型

| 类型 | 标签 | 行为 |
|---|---|---|
| `archive` | 归档 | 审批通过后标记为"已批准"，进入归档流程 |
| `payment` | 支付 | 审批通过后标记为"已批准"，进入支付流程 |
| 默认 | 无 | 普通审批节点 |

终节点类型由系统根据业务模块自动判断（如付款申请的最后一个节点自动为"支付"类型）。

### 3.4 自动跳过逻辑

对于项目级角色（项目经理、设计负责人/生产经理），如果审批人恰好就是该角色对应的用户（如项目经理自己发起的报销），系统会自动跳过该节点。

## 四、角色系统

### 4.1 系统内置角色

| 角色 | 角色码 | 是否可编辑 | 是否可删除 |
|---|---|---|---|
| 超级管理员 | `admin` | 不可编辑 | 不可删除 |
| 财务 | `finance` | 可编辑访问模块，不可修改名称 | 不可删除 |

### 4.2 默认角色列表

管理员在"角色设置"中首次初始化时，系统会创建以下角色：

| 角色 | 角色码 | 是否项目关联 |
|---|---|---|
| 部门负责人 | `dept_head` | 否 |
| 项目经理 | `project_manager` | 是 |
| 项目管理部 | `pmo` | 否 |
| 行政 | `admin_dept` | 否 |
| 采购部 | `procurement` | 否 |
| 设计负责人/生产经理 | `production` | 是 |
| 财务 | `finance` | 否 |
| 出纳 | `cashier` | 否 |
| 副总经理 | `vice_gm` | 否 |
| 总经理 | `gm` | 否 |
| 董事长 | `chairman` | 否 |

### 4.3 项目关联角色

项目关联角色（`isProjectRole = true`）的用户不是固定的，而是根据项目动态解析：

- **项目经理（`project_manager`）**：从项目的"项目经理"字段获取
- **设计负责人/生产经理（`production`）**：从项目的"设计负责人"或"生产经理"字段获取

## 五、管理员兜底功能

### 5.1 强制推进

当会签节点因人员缺席等原因卡住时，admin 角色用户可以点击橙色的"强制推进"按钮，跳过当前节点直接推进到下一节点。

**操作方式**：在审批详情页，admin 用户会看到橙色的"强制推进"按钮。

**API**：`POST /api/approval-flows/force-advance`

请求体：
```json
{
  "instanceId": "审批实例ID",
  "comment": "强制推进原因（可选）",
  "projectSourceId": "项目编号（可选）"
}
```

### 5.2 审批调试

管理员可通过"审批调试"页面（`/settings/approval-debug`）查看和修改审批状态。

## 六、发起权限校验

### 6.1 校验逻辑

用户提交审批时，系统会检查当前用户是否匹配流程第一节点的角色。不匹配则返回 403 错误。

### 6.2 校验函数

```typescript
canInitiateFlow({
  businessType: string,  // 业务模块类型
  flowLevel: string,     // 流程层级（project/company/common）
  userId: string,        // 当前用户ID
}): Promise<boolean>
```

### 6.3 已接入校验的 API

| API 路径 | 说明 |
|---|---|
| `POST /api/approval-instances` | 通用审批发起接口 |
| `POST /api/project-leads/bond-payment` | 借出款专用接口 |

## 七、前端组件集成

### 7.1 ApprovalActionButton

所有业务页面的审批操作按钮组件，支持以下 prop：

| Prop | 类型 | 说明 |
|---|---|---|
| `isAdmin` | `boolean` | 是否显示"强制推进"按钮 |
| `instanceId` | `string \| null` | 审批实例ID |
| `businessType` | `string` | 业务模块类型 |
| `currentStatus` | `string` | 当前审批状态 |
| 其他... | | 详见组件定义 |

**已集成 `isAdmin` 的页面**（共 8 个页面 12 处）：

- 费用报销（5 个实例）
- 采购需求（1 个实例）
- 采购询价（1 个实例）
- 采购送货（1 个实例）
- 支出合同（1 个实例）
- 收入合同（1 个实例）
- 非合同收支（1 个实例）
- 外包任务（1 个实例）

### 7.2 isAdmin 判断方式

```tsx
isAdmin={user?.roles?.some((r: any) => r.code === "admin") || false}
```

## 八、数据库模型

### 8.1 核心表

| 表名 | 说明 |
|---|---|
| `ApprovalFlowDefinition` | 审批流程定义（节点配置） |
| `ApprovalInstance` | 审批实例（每次发起生成一条） |
| `ApprovalAction` | 审批动作记录（通过/驳回/发起/跳过等） |

### 8.2 ApprovalFlowDefinition 关键字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `businessType` | String | 业务模块类型 |
| `flowLevel` | String | 流程层级（project/company/common） |
| `nodeOrder` | Int | 节点序号（从 1 开始） |
| `nodeName` | String | 节点名称（如"总经理审批"） |
| `approverRole` | String | 审批角色码，逗号分隔（如 `finance,cashier`） |
| `nodeType` | String | 节点类型（approval/archive/payment） |
| `isActive` | Boolean | 是否启用 |

### 8.3 ApprovalAction 动作类型

| 动作 | 说明 |
|---|---|
| `initiate` | 发起（第一节点自动记录） |
| `approve` | 审批通过 |
| `reject` | 驳回 |
| `archive` | 归档 |
| `payment` | 支付 |
| `auto_skip` | 自动跳过（如项目经理自己发起） |
| `force_advance` | 管理员强制推进 |

## 十、新增业务模块接入指南

当需要为新的业务模块接入审批流程时，按以下步骤操作：

### 步骤 1：在流程设置页面注册业务模块

在 `approval-flow/page.tsx` 的 `BUSINESS_MODULES` 数组中添加新的模块定义。

### 步骤 2：在业务页面集成审批组件

```tsx
import { ApprovalTimeline, ApprovalActionButton } from "@/components/ApprovalComponents";
import { useAuth } from "@/contexts/AuthContext";

export default function MyBusinessPage() {
  const { user } = useAuth();

  return (
    <>
      <ApprovalTimeline businessType="my_module" businessId={record.id} flowLevel="common" />
      <ApprovalActionButton
        instanceId={instanceId}
        businessType="my_module"
        businessId={record.id}
        flowLevel="common"
        currentStatus={record.status}
        isAdmin={user?.roles?.some((r: any) => r.code === "admin") || false}
        onStatusChange={(newStatus, newInstanceId) => { /* 更新状态 */ }}
      />
    </>
  );
}
```

### 步骤 3：在提交审批 API 中添加权限校验

```typescript
import { startApprovalFlow, canInitiateFlow } from "@/lib/approval-engine";

// 发起审批前校验
const canInitiate = await canInitiateFlow({
  businessType: "my_module",
  flowLevel: "common",
  userId: user.id,
});
if (!canInitiate) {
  return NextResponse.json({ error: "您没有权限发起此审批流程" }, { status: 403 });
}

// 发起审批
await startApprovalFlow({
  businessType: "my_module",
  businessId: record.id,
  flowLevel: "common",
  initiatorId: user.id,
});
```

### 步骤 4：管理员在流程设置界面配置节点和角色

无需修改代码，管理员在界面上完成配置即可生效。
