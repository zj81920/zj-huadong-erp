# 财务支出与收入详情优化设计文档

> 日期：2026-06-02 | 状态：已确认

---

## 一、概述

### 背景

当前财务支出/收入模块存在以下问题：
1. 支付申请时没有体现对方银行账户信息
2. 合同支付流程中金额显示过于紧凑
3. 没有体现已收票金额
4. 详情/记录功能分散在多个按钮，操作入口不统一

### 目标

统一所有财务支出/收入类型的详情展示风格，补充银行账户信息流，优化信息密度和用户体验。

---

## 二、数据库层变更

### 2.1 新增往来信息表（CounterpartyInfo）

```prisma
model CounterpartyInfo {
  id          String   @id @default(cuid())
  name        String   // 交易对方名称
  bankName    String?  @map("bank_name")  // 开户行
  bankAccount String?  @map("bank_account") // 银行账号
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")

  @@map("counterparty_infos")
}
```

**唯一性规则**：name + bankName + bankAccount 三者完全一致才视为同一条记录，任一不同即为新记录。

### 2.2 PaymentApplication 表字段使用

`PaymentApplication.bankAccount` 字段已存在于 schema 中（第821行），当前前端未使用，本次将其激活使用。

### 2.3 已有字段利用

- `ExpenseContract.invoicedAmount` — 已有字段，前端接口需扩展接收
- `Supplier.bankName` / `Supplier.bankAccount` — 已有字段，前端接口需扩展接收
- `User.bankName` / `User.bankAccount` — 已有字段，费用报销时自动带入

## 四、合同支出列表表格重设计

### 4.1 表格列

| 列序号 | 列名 | 数据来源 |
|-------|------|---------|
| 1 | 项目 | sourceContract.project.name / projectSourceId |

## 五、统一详情弹窗（所有类型通用模板）

### 5.1 弹窗布局

```
┌─────────────────────────────────────────────┐
│  弹窗标题                                    │
├─────────────────────────────────────────────┤
│  📄 [来源编号]  |  🏢 [对方名称]             │  ← 第一层：基本信息
│  关联项目: [项目名称] ([项目ID])              │
├─────────────────────────────────────────────┤
│  金额标签1      金额标签2      金额标签3       │  ← 第二层：金额网格
│  ¥xxx,xxx.xx  ¥xxx,xxx.xx  ¥xxx,xxx.xx   │     2-4列根据类型动态调整
├─────────────────────────────────────────────┤
│  🏦 收款方银行信息                            │  ← 第三层：银行信息
│  开户行：[银行名称]                           │     来源根据类型不同
│  账  号：[脱敏账号]                           │
├─────────────────────────────────────────────┤
│  付款申请记录 + 审批时间线                     │  ← 第四层：付款记录
│  （仅在合同支出/合同收入类型中显示）            │
└─────────────────────────────────────────────┘
```

### 5.2 各类型详情弹窗差异

| 类型 | 金额网格内容 | 银行信息来源 | 是否含付款记录 |
|------|------------|-------------|--------------|
| **合同支出** | 合同金额 / 已收票 / 已付 / 未付 | 供应商库自动带出（只读） | ✅ |
| **合同收入** | 合同金额 / 已开票 / 已收 / 未收 | 客户库自动带出（只读） | ✅ |
| **其他支出** | 支出金额（单值） | 手动填写 / 往来信息库 | ❌ |
| **借出款** | 借出金额 / 已归还 / 未还 | 手动填写 / 往来信息库 | ❌ |
| **费用报销** | 报销金额（单值） | 用户档案自动带出（只读） | ❌ |
| **借入资金归还** | 归还金额（单值） | 手动填写 / 往来信息库 | ❌ |
| **股东出资（收入）** | 出资金额（单值） | 手动填写 | ❌ |
| **其他收入** | 收入金额（单值） | 手动填写 | ❌ |

### 5.3 金额网格样式规范

- 使用 CSS Grid `grid-cols-2` 或 `grid-cols-4` 布局
- 每个格子包含标签（上）+ 金额（下）
- 标签字体：`text-[12px] text-[#78716C]`
- 金额字体：`text-[17px] font-bold text-[#1C1917]`
- 网格背景：`bg-[#FAFAF9] rounded-xl p-4`

## 六、付款申请弹窗

### 6.1 与详情弹窗的区别

| 弹窗 | 触发按钮 | 作用 | 审批时间线 |
|------|---------|------|-----------|
| **详情** | "详情" | 查看全貌（卡片+记录+审批） | ✅ 有 |
| **付款申请** | "付款" | 创建新申请（表单） | ❌ 无（尚未提交） |

### 6.2 付款申请弹窗卡片化

选择应付记录后，展示信息卡片（合同编号+供应商+金额网格+银行信息），下方保留表单字段：
- 本次付款金额
- 付款事由
- 申请人
- 备注

## 七、银行账号修改权限

### 7.1 管理员编辑银行账号

| 申请状态 | 管理员可修改 | 方式 |
|---------|------------|------|
| **草稿** | ✅ | 详情弹窗中编辑，与原申请人编辑逻辑一致 |
| **审批中** | ✅ | 详情弹窗中直接修改 bankAccount 字段，不影响审批流程 |
| **已批准** | ✅ | 同上 |
| **已驳回** | ✅ | 重新编辑提交 |

**核心原则**：银行账号属于支付信息修正，不涉及金额变更，不影响审批结论。管理员修改后仅更新 `bankAccount` 字段，审批状态保持不变。

### 7.2 同步到往来信息库

管理员修改银行账号后，自动将新的（名称+开户行+账号）保存到往来信息库（如果不存在）。

## 八、往来信息库

### 8.1 系统设置菜单

在侧边栏"系统设置"组新增菜单项：

```
系统设置
├── 部门设置
├── 角色设置
├── 用户设置
├── 流程设置
├── 往来信息管理    ← 新增
├── 审批调试
└── AI 模型配置
```

路由：`/settings/counterparty`

权限：仅管理员可见

### 8.2 功能

- 列表展示所有往来信息（名称、开户行、账号、创建时间）
- 可搜索/筛选
- 增删改查
- 可手动新增，也可由系统自动保存（首次使用时）

### 8.3 自动匹配规则

在"其他支出"和"借出款"表单中：

1. 用户输入"交易对方/借入方"名称
2. 前端实时匹配往来信息库中同名记录
3. 显示下拉建议，用户选中后自动填充开户行和账号
4. 用户提交后，如该组合（名称+开户行+账号）在库中不存在，自动保存

## 九、投标保证金发起弹窗变更

### 9.1 当前状态

项目线索详情页（`business/project-leads/[id]/BiddingSection.tsx`）中"发起保证金支付"弹窗仅展示保证金金额和确认按钮。

### 9.2 变更内容

在弹窗中新增三个字段：

```
交易对方名称 *  [输入时匹配往来信息库，自动带出客户名称]  
开户行        [自动填充，可修改]
银行账号      [自动填充，可修改]
```

提交时：
1. 将银行信息传给 `POST /api/project-leads/bond-payment`
2. API 端将这些信息存入借出款记录
3. 自动将（名称+开户行+账号）保存到往来信息库
